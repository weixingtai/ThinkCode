# Android MVP 架构

## 一、Android MVP架构详解
MVP（Model-View-Presenter）是在MVC基础上优化的架构模式，核心是**彻底解耦View和Model**，通过Presenter作为中间层隔离二者，解决了MVC中Activity/Fragment既当View又当Controller导致的职责臃肿问题。

### 1. 各模块定义与Android中的对应实现
| 模块             | 核心职责                                                 | Android 典型对应组件                                                    |
|----------------|------------------------------------------------------|-------------------------------------------------------------------|
| Model（模型）      | 与MVC中Model一致，处理数据逻辑（获取、存储、业务处理），不依赖View和Presenter    | 实体类（Bean）、Repository、网络请求（Retrofit）、数据库操作（Room）、业务逻辑类             |
| View（视图）       | 仅负责展示数据、接收用户交互，所有业务逻辑委托给Presenter，不直接接触Model         | Activity/Fragment、Layout布局、UI控件（TextView/Button等）                 |
| Presenter（主持人） | 作为View和Model的桥梁，接收View的交互事件，调用Model处理数据，处理完后通知View更新 | 独立的Presenter类（如LoginPresenter），持有View的接口引用，不直接持有Activity/Fragment |

### 2. MVP核心设计原则
- **View与Model完全解耦**：View只负责UI，Model只负责数据，二者无任何直接交互；
- **面向接口编程**：通常为View定义接口（如ILoginView），Presenter持有接口而非具体View实例，降低耦合；
- **Presenter可复用**：同一个Presenter可对接不同View（如手机端和平板端的登录View）。

### 4. 完整示例（Android MVP实现登录功能）
##### 步骤1：定义View接口（解耦Presenter与具体View）
```kotlin
// ILoginView.kt：View的行为抽象，Presenter通过接口操作View
interface ILoginView {
    // 显示加载中
    fun showLoading()
    // 隐藏加载中
    fun hideLoading()
    // 登录成功回调
    fun onLoginSuccess(msg: String)
    // 登录失败回调
    fun onLoginFail(msg: String)
    // 获取输入的用户名
    fun getUsername(): String
    // 获取输入的密码
    fun getPassword(): String
}
```

##### 步骤2：实现Model层（数据逻辑）
```kotlin
// LoginModel.kt：仅处理数据，无任何UI依赖
class LoginModel {
    // 模拟网络登录请求
    fun login(username: String, password: String, callback: (Boolean, String) -> Unit) {
        Thread {
            Thread.sleep(1000) // 模拟网络延迟
            if (username == "admin" && password == "123456") {
                callback(true, "登录成功")
            } else {
                callback(false, "用户名或密码错误")
            }
        }.start()
    }
}
```

##### 步骤3：实现Presenter层（中间桥梁）
```kotlin
// LoginPresenter.kt：连接View和Model，处理业务逻辑
class LoginPresenter(private val loginView: ILoginView) {
    private val loginModel = LoginModel()

    // 处理登录逻辑
    fun login() {
        // 通知View显示加载中
        loginView.showLoading()
        val username = loginView.getUsername()
        val password = loginView.getPassword()
        
        // 调用Model处理数据
        loginModel.login(username, password) { isSuccess, msg ->
            // 通知View隐藏加载中
            loginView.hideLoading()
            if (isSuccess) {
                loginView.onLoginSuccess(msg)
            } else {
                loginView.onLoginFail(msg)
            }
        }
    }

    // 防止内存泄漏，解绑View
    fun detachView() {
        // 可清空回调、取消网络请求等
    }
}
```

##### 步骤4：实现View层（Activity）
```kotlin
// LoginActivity.kt：仅负责UI展示和交互，业务逻辑交给Presenter
class LoginActivity : AppCompatActivity(), ILoginView {
    private lateinit var etUsername: EditText
    private lateinit var etPassword: EditText
    private lateinit var btnLogin: Button
    private lateinit var progressBar: ProgressBar
    private lateinit var loginPresenter: LoginPresenter

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_login)
        
        // 初始化UI控件
        etUsername = findViewById(R.id.et_username)
        etPassword = findViewById(R.id.et_password)
        btnLogin = findViewById(R.id.btn_login)
        progressBar = findViewById(R.id.progress_bar)
        
        // 创建Presenter，传入View接口
        loginPresenter = LoginPresenter(this)
        
        // 处理用户交互：仅调用Presenter方法，无业务逻辑
        btnLogin.setOnClickListener {
            loginPresenter.login()
        }
    }

    // 实现ILoginView接口方法
    override fun showLoading() {
        progressBar.visibility = View.VISIBLE
        btnLogin.isEnabled = false
    }

    override fun hideLoading() {
        progressBar.visibility = View.GONE
        btnLogin.isEnabled = true
    }

    override fun onLoginSuccess(msg: String) {
        Toast.makeText(this, msg, Toast.LENGTH_SHORT).show()
        startActivity(Intent(this, HomeActivity::class.java))
    }

    override fun onLoginFail(msg: String) {
        Toast.makeText(this, msg, Toast.LENGTH_SHORT).show()
    }

    override fun getUsername(): String {
        return etUsername.text.toString()
    }

    override fun getPassword(): String {
        return etPassword.text.toString()
    }

    // 生命周期管理：防止内存泄漏
    override fun onDestroy() {
        super.onDestroy()
        loginPresenter.detachView()
    }
}
```

## 二、MVP架构 vs 未使用架构（混乱代码）
未使用架构的代码仍以“万能Activity”为核心，二者核心区别如下：

| 维度     | MVP架构                                              | 未使用架构（混乱代码）                         |
|--------|----------------------------------------------------|-------------------------------------|
| 代码组织   | 按职责严格拆分：View（UI）、Presenter（逻辑）、Model（数据），层与层通过接口交互 | 所有逻辑（UI、数据、业务）堆砌在Activity/Fragment中 |
| 耦合度    | View与Model完全解耦，Presenter与View通过接口弱耦合               | View与Model强耦合，改数据逻辑必动UI代码           |
| 可读性    | 模块边界清晰：改登录逻辑找Presenter，改UI找Activity，改数据找Model      | 代码冗长，一个功能分散在几百行代码中                  |
| 可维护性   | 单模块修改不影响其他模块（如改加载动画仅动View）                         | 改一个小功能可能触发连锁问题（如改密码规则影响UI）          |
| 可复用性   | Presenter可复用（如登录逻辑适配手机/平板View）、Model可跨页面复用         | 逻辑与UI绑定，无法复用，重复写相同代码                |
| 可测试性   | Presenter可脱离Android环境单元测试（Mock View接口），Model可单独测试  | 需启动应用、手动操作UI才能测试，效率极低               |
| 内存泄漏风险 | 可通过detachView主动解绑，风险低                              | 匿名内部类、线程持有Activity引用，易内存泄漏          |

### 未使用架构的反面示例（对比参考）
```kotlin
// 所有逻辑堆砌，无拆分
class BadLoginActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_login)
        val etUsername = findViewById<EditText>(R.id.et_username)
        val etPassword = findViewById<EditText>(R.id.et_password)
        val btnLogin = findViewById<Button>(R.id.btn_login)

        btnLogin.setOnClickListener {
            val username = etUsername.text.toString()
            val password = etPassword.text.toString()
            // 数据逻辑、UI逻辑、网络请求全混在一起
            Thread {
                Thread.sleep(1000)
                val isSuccess = username == "admin" && password == "123456"
                runOnUiThread {
                    if (isSuccess) {
                        Toast.makeText(this, "登录成功", Toast.LENGTH_SHORT).show()
                    } else {
                        Toast.makeText(this, "登录失败", Toast.LENGTH_SHORT).show()
                    }
                }
            }.start()
        }
    }
}
```

## 三、MVP架构的使用场景
MVP是Android中**中阶、高解耦**的架构模式，平衡了开发成本和架构规范性，适合以下场景：

### 1. 优先使用MVP的场景
- **中小型商业App**：如资讯类、工具类+中等复杂度功能（带登录、列表、详情），既解决MVC的臃肿问题，又无需MVVM的学习成本；
- **需要高可测试性的项目**：如企业级应用、金融类App（需单元测试保障稳定性），MVP的Presenter可脱离Android环境测试；
- **团队协作开发**：多人维护同一模块（如登录模块由A写UI、B写逻辑），接口化设计减少协作冲突；
- **MVC项目重构**：原有MVC项目出现Activity臃肿、维护困难，重构为MVP成本低（仅拆分Presenter），风险可控。

### 2. 谨慎使用/不推荐MVP的场景
- **超小型App/原型开发**：如单页面计算器、记事本，MVP的接口和类拆分增加开发量，不如MVC高效；
- **大型复杂App（带数据流管理）**：如电商、社交App（需状态管理、数据缓存），MVP的Presenter易出现“逻辑堆积”，建议用MVVM+Jetpack；
- **新手入门项目**：MVP的接口、解耦概念对纯新手较抽象，建议先学MVC建立基础认知。

## 总结
1. **MVP核心**：通过Presenter层彻底隔离View和Model，View仅负责UI、Presenter处理业务逻辑、Model处理数据，层与层通过接口交互，解决了MVC中Activity职责臃肿的问题；
2. **MVP vs 无架构**：MVP胜在低耦合、高可测试性、易维护，无架构代码则混乱耦合，仅适合极简单场景；
3. **MVP适用场景**：中小型商业App、需要高可测试性的项目、团队协作开发、MVC项目重构；超小型App或大型复杂App不推荐。
