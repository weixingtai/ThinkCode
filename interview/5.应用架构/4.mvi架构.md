# Android MVI 架构

## 一、Android MVI架构详解
MVI（Model-View-Intent）是基于**单向数据流**设计的响应式架构模式，核心是将UI的所有状态抽象为`State`，用户交互抽象为`Intent`，数据和状态流转全程单向、可追溯，解决了MVVM中数据流分散、状态难以追踪的问题，是比MVVM更严格的响应式架构。

### 1. 各模块定义与Android中的对应实现
| 模块         | 核心职责                                                       | Android 典型对应组件                                                                    |
|------------|------------------------------------------------------------|-----------------------------------------------------------------------------------|
| Model（模型）  | 处理数据逻辑（获取、存储、业务处理），接收`Intent`并返回`State`，无UI依赖              | 实体类（Bean）、Repository、网络请求（Retrofit）、数据库操作（Room）、业务逻辑类                             |
| View（视图）   | 展示`State`数据、将用户交互封装为`Intent`发送给ViewModel，无业务逻辑             | Activity/Fragment、Layout布局、UI控件（TextView/Button等）、ViewBinding                     |
| Intent（意图） | 封装用户所有交互行为（点击、输入、刷新等），是View向ViewModel发送的唯一指令               | 密封类（Sealed Class）/枚举，如`LoginIntent`（Login、ClearInput、Retry等）                      |
| State（状态）  | 封装View的所有UI状态（加载中、成功、失败、空数据等），是ViewModel向View输出的唯一数据       | 数据类（Data Class），如`LoginState`（loading: Boolean, success: String?, error: String?） |
| ViewModel  | 接收`Intent`，调用Model处理数据，将结果转换为`State`通过Flow/LiveData发送给View | 继承`ViewModel`的类，内部通过`SharedFlow`/`StateFlow`暴露`State`                             |

### 2. MVI核心设计原则
- **单向数据流**：`View → Intent → ViewModel → Model → State → View`，全程单向，无反向依赖；
- **状态唯一可信源**：View的所有UI状态都由`State`驱动，不存在“局部更新UI”，状态可预测；
- **不可变状态**：`State`设计为不可变对象（val），状态更新通过生成新`State`实现，避免并发问题；
- **意图标准化**：所有用户交互都封装为`Intent`，便于追踪、测试和复现问题。

// LoginRepository.kt：数据仓库
class LoginRepository {
    // 模拟网络登录请求
    suspend fun login(username: String, password: String): Result<String> {
        delay(1000) // 模拟网络延迟
        return if (username == "admin" && password == "123456") {
            Result.success("登录成功")
        } else {
            Result.failure(IllegalArgumentException("用户名或密码错误"))
        }
    }
}
```

#### 步骤4：实现ViewModel层（核心，处理Intent并输出State）
```kotlin
// LoginViewModel.kt：MVI核心，单向数据流处理
class LoginViewModel : ViewModel() {
    // 接收Intent的Flow（SharedFlow支持多订阅者）
    private val _intent = MutableSharedFlow<LoginIntent>()
    // 输出State的Flow（StateFlow默认值+防抖）
    private val _state = MutableStateFlow(LoginState())
    val state: StateFlow<LoginState> = _state

    private val repository = LoginRepository()

    init {
        // 监听Intent并处理
        handleIntent()
    }

    // 发送Intent的方法（View调用）
    fun sendIntent(intent: LoginIntent) {
        viewModelScope.launch {
            _intent.emit(intent)
        }
    }

    // 处理Intent，转换为State
    private fun handleIntent() {
        viewModelScope.launch {
            _intent.collect { intent ->
                when (intent) {
                    is LoginIntent.Login -> handleLogin(intent.username, intent.password)
                    LoginIntent.ClearInput -> handleClearInput()
                    LoginIntent.Retry -> handleRetry()
                }
            }
        }
    }

    // 处理登录Intent
    private suspend fun handleLogin(username: String, password: String) {
        // 1. 发送加载中State
        _state.emit(_state.value.copy(isLoading = true, loginSuccess = null, loginError = null))
        // 2. 调用Model处理数据
        val result = repository.login(username, password)
        // 3. 根据结果生成新State
        val newState = when (result) {
            is Result.Success -> _state.value.copy(
                isLoading = false,
                loginSuccess = result.data,
                loginError = null
            )
            is Result.Failure -> _state.value.copy(
                isLoading = false,
                loginSuccess = null,
                loginError = result.exception.message ?: "登录失败"
            )
        }
        // 4. 发送新State
        _state.emit(newState)
    }

    // 处理清空输入Intent
    private suspend fun handleClearInput() {
        _state.emit(_state.value.copy(isInputClear = true))
        // 重置清空状态（避免重复清空）
        delay(100)
        _state.emit(_state.value.copy(isInputClear = false))
    }

    // 处理重试Intent
    private suspend fun handleRetry() {
        // 复用登录逻辑（需缓存上次输入的用户名/密码，此处简化）
        handleLogin("", "")
    }
}
```

#### 步骤5：实现View层（Activity）
```kotlin
// LoginActivity.kt：仅接收State并更新UI，发送Intent
class LoginActivity : AppCompatActivity() {
    private lateinit var binding: ActivityLoginBinding
    private lateinit var loginViewModel: LoginViewModel

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        binding = ActivityLoginBinding.inflate(layoutInflater)
        setContentView(binding.root)

        // 初始化ViewModel
        loginViewModel = ViewModelProvider(this)[LoginViewModel::class.java]

        // 1. 监听State变化，更新UI
        observeState()

        // 2. 绑定用户交互，发送Intent
        bindIntent()
    }

    // 监听State，更新UI（唯一的UI更新入口）
    private fun observeState() {
        lifecycleScope.launch {
            repeatOnLifecycle(Lifecycle.State.STARTED) {
                loginViewModel.state.collect { state ->
                    // 更新加载状态
                    binding.progressBar.visibility = if (state.isLoading) View.VISIBLE else View.GONE
                    binding.btnLogin.isEnabled = !state.isLoading

                    // 处理登录成功
                    state.loginSuccess?.let {
                        Toast.makeText(this@LoginActivity, it, Toast.LENGTH_SHORT).show()
                        startActivity(Intent(this@LoginActivity, HomeActivity::class.java))
                    }

                    // 处理登录失败
                    state.loginError?.let {
                        Toast.makeText(this@LoginActivity, it, Toast.LENGTH_SHORT).show()
                    }

                    // 处理清空输入
                    if (state.isInputClear) {
                        binding.etUsername.setText("")
                        binding.etPassword.setText("")
                    }
                }
            }
        }
    }

    // 绑定用户交互，发送Intent（唯一的Intent发送入口）
    private fun bindIntent() {
        // 登录按钮
        binding.btnLogin.setOnClickListener {
            val username = binding.etUsername.text.toString()
            val password = binding.etPassword.text.toString()
            loginViewModel.sendIntent(LoginIntent.Login(username, password))
        }

        // 清空按钮
        binding.btnClear.setOnClickListener {
            loginViewModel.sendIntent(LoginIntent.ClearInput)
        }

        // 重试按钮
        binding.btnRetry.setOnClickListener {
            loginViewModel.sendIntent(LoginIntent.Retry)
        }
    }
}
```

## 二、MVI架构 vs 未使用架构（混乱代码）
| 维度    | MVI架构                                                 | 未使用架构（混乱代码）                               |
|-------|-------------------------------------------------------|-------------------------------------------|
| 代码组织  | 严格单向数据流：View→Intent→ViewModel→Model→State→View，模块职责唯一 | 所有逻辑（UI、数据、异步、状态）堆砌在Activity/Fragment     |
| 状态管理  | 单一不可变State驱动所有UI，状态可预测、可追溯                            | 零散更新UI（如手动改TextView文本、ProgressBar显示），状态混乱 |
| 数据流方向 | 全程单向，无反向依赖，易调试                                        | 数据流混乱（UI改数据、数据改UI、异步回调改UI），难定位问题          |
| 可测试性  | Intent/State标准化，ViewModel可纯函数式测试，无需启动UI               | 需手动操作UI测试，异步/状态问题难以复现                     |
| 可维护性  | 新增功能仅需扩展Intent/State，无连锁修改                            | 改一个功能需修改多处UI/数据逻辑，易引入bug                  |
| 错误复现  | 可通过Intent/State日志复现所有用户操作和状态变化                        | 无法追溯用户操作路径，复现bug依赖“手动还原步骤”                |

### 未使用架构的反面示例（对比参考）
```kotlin
// 状态混乱、数据流双向、无标准化
class BadLoginActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_login)
        val etUsername = findViewById<EditText>(R.id.et_username)
        val etPassword = findViewById<EditText>(R.id.et_password)
        val btnLogin = findViewById<Button>(R.id.btn_login)
        val progressBar = findViewById<ProgressBar>(R.id.progress_bar)

        btnLogin.setOnClickListener {
            progressBar.visibility = View.VISIBLE
            val username = etUsername.text.toString()
            val password = etPassword.text.toString()
            // 异步、数据、UI逻辑混在一起，状态零散更新
            Thread {
                Thread.sleep(1000)
                val isSuccess = username == "admin" && password == "123456"
                runOnUiThread {
                    progressBar.visibility = View.GONE
                    if (isSuccess) {
                        Toast.makeText(this, "登录成功", Toast.LENGTH_SHORT).show()
                    } else {
                        Toast.makeText(this, "登录失败", Toast.LENGTH_SHORT).show()
                        // 手动改按钮状态（零散更新）
                        btnLogin.text = "重新登录"
                    }
                }
            }.start()
        }
    }
}
```

## 三、MVI架构的使用场景
MVI是Android**高阶、响应式**架构模式，适合对状态可预测性、可维护性要求高的场景：

### 1. 优先使用MVI的场景
- **大型复杂App（高状态复杂度）**：如电商（购物车、订单状态）、社交（聊天、消息状态）、金融（交易、持仓状态），状态可预测性至关重要；
- **需要高可测试性/可追溯性的项目**：如医疗、金融级应用，需复现所有用户操作和状态变化，满足合规要求；
- **多人协作的大型团队项目**：标准化的Intent/State减少团队理解分歧，便于代码评审和维护；
- **实时性高的响应式UI**：如直播、股票、物联网数据展示，State可统一管理实时数据和UI状态；
- **MVVM架构升级**：MVVM项目出现状态分散、数据流混乱，MVI可规范数据流，提升可维护性。

### 2. 谨慎使用/不推荐MVI的场景
- **超小型App/快速原型**：如单页面计算器、记事本，MVI的Intent/State封装增加开发量，不如MVC/MVVM高效；
- **纯新手入门项目**：MVI涉及单向数据流、不可变状态、Flow等概念，学习曲线陡峭，建议先学MVVM；
- **简单CRUD应用**：如基础的增删改查工具，状态简单，MVI的规范带来的收益低于开发成本；
- **极致性能要求的场景**：MVI的State更新需生成新对象，有轻微开销（可忽略），游戏/实时渲染类App可简化使用。

## 总结
1. **MVI核心**：基于单向数据流，将用户交互封装为Intent、UI状态封装为不可变State，全程`View→Intent→ViewModel→Model→State→View`单向流转，状态可预测、可追溯；
2. **MVI vs 无架构**：MVI胜在状态统一、数据流可预测、易调试，无架构代码则状态混乱、数据流双向，仅适合极简单场景；
3. **MVI适用场景**：大型复杂App、高可测试性项目、多人协作团队、实时响应式UI；超小型App/新手项目可暂缓使用。
