# Android MVVM 架构

## 一、Android MVVM架构详解
MVVM（Model-View-ViewModel）是基于MV*架构演进的现代架构模式，核心是**数据驱动UI + 彻底解耦**，依托Android Jetpack（LiveData、ViewModel、DataBinding/ViewBinding）实现，解决了MVP中Presenter与View接口耦合、手动更新UI的问题，是目前Android官方推荐的架构模式。

### 1. 各模块定义与Android中的对应实现
| 模块              | 核心职责                                                         | Android 典型对应组件                                                                   |
|-----------------|--------------------------------------------------------------|----------------------------------------------------------------------------------|
| Model（模型）       | 与MVC/MVP一致，处理数据逻辑（获取、存储、业务处理），不依赖上层组件                        | 实体类（Bean）、Repository、网络请求（Retrofit）、数据库操作（Room）、业务逻辑类                            |
| View（视图）        | 展示数据、接收用户交互，通过DataBinding/ViewBinding绑定ViewModel数据，无业务逻辑     | Activity/Fragment、Layout布局（带DataBinding）、UI控件（TextView/Button等）                  |
| ViewModel（视图模型） | 作为View和Model的桥梁，持有UI相关数据（非UI对象），通过LiveData暴露可观察数据，处理UI相关业务逻辑 | 继承`ViewModel`/`AndroidViewModel`的类（如LoginViewModel），内部持有Model引用，通过LiveData通知View |
| DataBinding（可选） | 数据绑定框架，实现View与ViewModel的双向绑定，自动更新UI                          | 布局文件中的`<layout>`标签、@{}表达式、双向绑定@={}                                               |

### 2. MVVM核心设计原则
- **数据驱动UI**：View的更新由ViewModel的LiveData自动触发，无需手动调用View接口；
- **生命周期感知**：ViewModel与Activity/Fragment生命周期解耦（屏幕旋转不重建），LiveData自动感知生命周期避免内存泄漏；
- **双向绑定（可选）**：通过DataBinding实现UI输入自动同步到ViewModel，ViewModel数据变化自动更新UI；
- **无直接引用**：View持有ViewModel引用，但ViewModel完全不感知View，彻底解耦。

### 4. 完整示例（Android MVVM实现登录功能，基于Jetpack）
#### 前置条件
- 开启DataBinding：在module的`build.gradle`中配置
```gradle
android {
    ...
    buildFeatures {
        dataBinding true // 开启DataBinding
        viewBinding true // 可选，替代findViewById
    }
}
```

#### 步骤1：实现Model层（数据逻辑）
```kotlin
// LoginRepository.kt：数据仓库，统一管理数据来源（网络/本地）
class LoginRepository {
    // 模拟网络登录请求
    fun login(username: String, password: String): Flow<Result<String>> {
        return flow {
            delay(1000) // 模拟网络延迟
            if (username == "admin" && password == "123456") {
                emit(Result.success("登录成功"))
            } else {
                emit(Result.failure(IllegalArgumentException("用户名或密码错误")))
            }
        }.flowOn(Dispatchers.IO) // 切换到IO线程
    }
}

// 通用Result封装类
sealed class Result<out T> {
    data class Success<out T>(val data: T) : Result<T>()
    data class Failure(val exception: Throwable) : Result<Nothing>()
}
```

#### 步骤2：实现ViewModel层（核心，持有数据和业务逻辑）
```kotlin
// LoginViewModel.kt：关联Model，暴露可观察数据给View
class LoginViewModel : ViewModel() {
    // 可观察的登录状态（LiveData自动感知生命周期）
    private val _loginState = MutableLiveData<Result<String>>()
    val loginState: LiveData<Result<String>> = _loginState

    // 可观察的加载状态
    private val _isLoading = MutableLiveData<Boolean>(false)
    val isLoading: LiveData<Boolean> = _isLoading

    // 绑定UI输入的用户名/密码（双向绑定用）
    val username = MutableLiveData<String>("")
    val password = MutableLiveData<String>("")

    private val repository = LoginRepository()

    // 处理登录逻辑
    fun login() {
        val name = username.value ?: ""
        val pwd = password.value ?: ""
        if (name.isEmpty() || pwd.isEmpty()) {
            _loginState.value = Result.Failure(IllegalArgumentException("用户名/密码不能为空"))
            return
        }

        _isLoading.value = true
        // 协程处理异步请求（ViewModelScope自动跟随ViewModel生命周期）
        viewModelScope.launch {
            repository.login(name, pwd).collect { result ->
                _isLoading.value = false
                _loginState.value = result
            }
        }
    }
}
```

#### 步骤3：实现View层（Activity + DataBinding布局）
##### 布局文件（activity_login.xml，带DataBinding）
```xml
<?xml version="1.0" encoding="utf-8"?>
<layout xmlns:android="http://schemas.android.com/apk/res/android">
    <!-- 数据绑定的变量声明 -->
    <LinearLayout
        android:layout_width="match_parent"
        android:layout_height="match_parent"
        android:orientation="vertical"
        android:padding="20dp">

<!-- 双向绑定：UI输入自动同步到viewModel.username -->
        <EditText
            android:layout_width="match_parent"
            android:layout_height="wrap_content"
            android:hint="请输入用户名"
            android:text="@={viewModel.username}" />

        <!-- 双向绑定：UI输入自动同步到viewModel.password -->
        <EditText
            android:layout_width="match_parent"
            android:layout_height="wrap_content"
            android:hint="请输入密码"
            android:inputType="textPassword"
            android:text="@={viewModel.password}" />

        <Button
            android:layout_width="match_parent"
            android:layout_height="wrap_content"
            android:text="登录"
            android:onClick="@{() -> viewModel.login()}"
            android:enabled="@{!viewModel.isLoading}" />

        <!-- 根据加载状态显示/隐藏ProgressBar -->
        <ProgressBar
            android:layout_width="wrap_content"
            android:layout_height="wrap_content"
            android:layout_gravity="center"
            android:visibility="@{viewModel.isLoading ? View.VISIBLE : View.GONE}" />
    </LinearLayout>
</layout>
```

##### Activity代码（LoginActivity.kt）
```kotlin
class LoginActivity : AppCompatActivity() {
    private lateinit var binding: ActivityLoginBinding
    private lateinit var loginViewModel: LoginViewModel

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        // 初始化DataBinding（替代setContentView）
        binding = DataBindingUtil.setContentView(this, R.layout.activity_login)
        // 初始化ViewModel（ViewModelProvider保证生命周期安全）
        loginViewModel = ViewModelProvider(this)[LoginViewModel::class.java]
        // 绑定ViewModel到布局
        binding.viewModel = loginViewModel
        // 设置生命周期所有者（让LiveData感知Activity生命周期）
        binding.lifecycleOwner = this

        // 观察登录状态变化（也可通过DataBinding直接绑定到UI）
        loginViewModel.loginState.observe(this) { result ->
            when (result) {
                is Result.Success -> {
                    Toast.makeText(this, result.data, Toast.LENGTH_SHORT).show()
                    startActivity(Intent(this, HomeActivity::class.java))
                }
                is Result.Failure -> {
                    Toast.makeText(this, result.exception.message, Toast.LENGTH_SHORT).show()
                }
            }
        }
    }
}
```

## 二、MVVM架构 vs 未使用架构（混乱代码）
| 维度     | MVVM架构                                                            | 未使用架构（混乱代码）                           |
|--------|-------------------------------------------------------------------|---------------------------------------|
| 代码组织   | 严格分层：Model（数据）、ViewModel（逻辑+数据）、View（UI），通过LiveData/DataBinding解耦 | 所有逻辑（UI、数据、业务、异步）堆砌在Activity/Fragment |
| 耦合度    | View与Model完全解耦，ViewModel不感知View，层与层通过可观察数据交互                      | View与Model强耦合，改数据必动UI代码               |
| UI更新方式 | 数据驱动：LiveData自动推送，DataBinding自动更新UI，无手动更新代码                       | 手动调用findViewById更新UI，易遗漏/出错           |
| 生命周期管理 | ViewModel独立于Activity生命周期（旋转不重建），LiveData自动解绑                      | 屏幕旋转重建数据丢失，易因生命周期错误导致内存泄漏             |
| 可维护性   | 单模块修改不影响其他模块（如改登录逻辑仅动ViewModel）                                   | 改一个功能需修改几百行混合代码，易触发连锁问题               |
| 可测试性   | ViewModel可脱离Android环境单元测试（Mock Repository），无需启动UI                 | 需手动操作UI测试，效率极低                        |
| 异步处理   | 基于ViewModelScope协程，自动取消异步任务，无内存泄漏                                 | 手动管理线程/回调，易出现异步任务泄漏                   |

### 未使用架构的反面示例（对比参考）
```kotlin
// 所有逻辑堆砌，无分层、无生命周期管理
class BadLoginActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_login)
        val etUsername = findViewById<EditText>(R.id.et_username)
        val etPassword = findViewById<EditText>(R.id.et_password)
        val btnLogin = findViewById<Button>(R.id.btn_login)

        btnLogin.setOnClickListener {
            val username = etUsername.text.toString()
            val password = etPassword.text.toString()
            // 异步、数据、UI逻辑混在一起
            Thread {
                Thread.sleep(1000)
                val isSuccess = username == "admin" && password == "123456"
                runOnUiThread {
                    if (isSuccess) {
                        Toast.makeText(this, "登录成功", Toast.LENGTH_SHORT).show()
                    } else {
                        Toast.makeText(this, "登录失败", Toast.LENGTH_SHORT).show()
                    }
                }
            }.start()
        }
    }
}
```

## 三、MVVM架构的使用场景
MVVM是Android**高阶、现代化**的架构模式，依托Jetpack生态，适合以下场景：

### 1. 优先使用MVVM的场景
- **中大型商业App**：如电商、社交、金融、资讯类App（需状态管理、数据缓存、多页面交互），MVVM的数据流管理更清晰；
- **需要高稳定性/低维护成本的项目**：如企业级应用、长期维护的产品，MVVM的低耦合、生命周期安全特性减少线上问题；
- **团队协作的大型项目**：多人分工（A写Repository、B写ViewModel、C写UI），模块边界清晰，减少协作冲突；
- **需要响应式UI的场景**：如实时数据展示（股票、聊天、直播），LiveData/Flow可高效实现数据实时更新；
- **MVP/MVC项目升级**：原有架构出现Presenter/Controller臃肿、手动更新UI繁琐，升级为MVVM可简化逻辑。

### 2. 谨慎使用/不推荐MVVM的场景
- **超小型App/快速原型**：如单页面计算器、记事本，MVVM的Jetpack配置、DataBinding学习成本增加开发量，不如MVC高效；
- **纯新手入门项目**：MVVM涉及协程、LiveData、DataBinding等多个概念，新手易混淆，建议先学MVC/MVP建立基础；
- **极低性能要求的场景**：DataBinding有轻微性能开销（可忽略），若需极致性能（如游戏、实时渲染），可简化为MVVM+ViewBinding（放弃DataBinding）。

## 总结
1. **MVVM核心**：基于数据驱动UI，通过ViewModel隔离View和Model，依托LiveData（生命周期感知）、DataBinding（自动更新UI）实现彻底解耦，是Android官方推荐架构；
2. **MVVM vs 无架构**：MVVM胜在低耦合、生命周期安全、数据驱动UI、易维护，无架构代码则混乱耦合，仅适合极简单场景；
3. **MVVM适用场景**：中大型商业App、高稳定性项目、团队协作项目、响应式UI场景；超小型App/纯新手项目可暂缓使用。
